# Documentation

Write for the reader's next action. Start with what the thing is, when
to use it, and the shortest working example. Move details and edge cases
after the main path.

Keep instructions testable. Use commands and examples that match the
repository's actual tooling. Avoid promising behavior the code does not
support.

Prefer clear headings, small examples, and explicit prerequisites. When
editing existing docs, preserve the local voice and structure unless it
is actively getting in the way.
